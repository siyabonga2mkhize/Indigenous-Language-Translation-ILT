## 1. Introduction

### 1.1 Purpose

This document outlines the database design approach for the **Indigenous Language Phrasebook & Translation Portal**. It serves as the planning foundation before creating the conceptual and logical ER diagrams. The document identifies all entities, their relationships, business rules, indexing needs, and data constraints based on the requirements defined in the SRS.

### 1.2 Scope

This plan covers the entire database schema for Version 1.0 of the application, including:

- Custom tables for phrases, translations, categories, favourites, submissions, and statistics
- Integration with ASP.NET Identity tables
- Indexing strategy for performance optimisation
- Data integrity rules and business constraints

## 3. Entity Definitions & Attributes

### 3.1 Category

**Description:** Organises phrases into campus-specific topics.

| Attribute | Data Type | Nullable | Description |
|-----------|-----------|----------|-------------|
| CategoryId | int | No | Primary key (identity) |
| Name | nvarchar(100) | No | Category name (e.g., "Registration") – must be unique |
| Description | nvarchar(500) | Yes | Optional category description |
| IsActive | bit | No | Soft delete flag (default: 1) |
| CreatedDate | datetime2 | No | Timestamp when record was created (default: GETDATE()) |
| ModifiedDate | datetime2 | Yes | Timestamp of last update |

**Business Rules:**
- Category names must be unique
- Only active categories can be assigned to phrases
- Deactivating a category doesn't delete its phrases but prevents new assignments
- Categories can be reactivated

**Predefined Categories:**
1. Registration
2. Accommodation
3. Health Services
4. Library
5. Academic Support

---

### 3.2 Phrase

**Description:** The core content entity – campus phrases in English.

| Attribute | Data Type | Nullable | Description |
|-----------|-----------|----------|-------------|
| PhraseId | int | No | Primary key (identity) |
| EnglishText | nvarchar(1000) | No | The phrase in English |
| CategoryId | int | No | Foreign key → Category.CategoryId |
| IsActive | bit | No | Soft delete flag (default: 1) |
| CreatedDate | datetime2 | No | Timestamp when record was created (default: GETDATE()) |
| ModifiedDate | datetime2 | Yes | Timestamp of last update |

**Business Rules:**
- Each phrase belongs to exactly one category
- EnglishText must be unique (or at least checked for duplicates to avoid confusion)
- Only active phrases appear in search results
- A phrase can have multiple translations (one per language)
- A phrase can be favourited by multiple users

---

### 3.3 Translation

**Description:** Translations of phrases in South African languages.

| Attribute | Data Type | Nullable | Description |
|-----------|-----------|----------|-------------|
| TranslationId | int | No | Primary key (identity) |
| PhraseId | int | No | Foreign key → Phrase.PhraseId |
| Language | nvarchar(50) | No | Target language (e.g., "isiZulu") |
| Text | nvarchar(2000) | No | The translated text |
| Status | nvarchar(20) | No | "Pending", "Approved", "Rejected" |
| RejectionReason | nvarchar(500) | Yes | Reason if rejected |
| SubmittedBy | nvarchar(450) | No | Foreign key → AspNetUsers.Id |
| ReviewedBy | nvarchar(450) | Yes | Foreign key → AspNetUsers.Id |
| SubmittedDate | datetime2 | No | When submitted (default: GETDATE()) |
| ReviewedDate | datetime2 | Yes | When reviewed |

**Business Rules:**
- Only one "Approved" translation per Phrase per Language
- Pending translations are hidden from search results
- Only Administrator can change status from Pending to Approved/Rejected
- Rejection reason is required when rejecting
- Translations are submitted by Students, reviewed by Administrators

**Supported Languages (11 Official):**
1. isiZulu
2. isiXhosa
3. Afrikaans
4. English (source only)
5. Sepedi
6. Sesotho
7. Setswana
8. Siswati
9. Tshivenda
10. Xitsonga
11. isiNdebele

---

### 3.4 Favourite

**Description:** Junction table linking Users to their favourite Phrases.

| Attribute | Data Type | Nullable | Description |
|-----------|-----------|----------|-------------|
| FavouriteId | int | No | Primary key (identity) |
| UserId | nvarchar(450) | No | Foreign key → AspNetUsers.Id |
| PhraseId | int | No | Foreign key → Phrase.PhraseId |
| AddedDate | datetime2 | No | When favourited (default: GETDATE()) |

**Business Rules:**
- A user can favourite a phrase only once (unique constraint on UserId + PhraseId)
- Only active phrases can be favourited
- Favourites persist across sessions
- Removing a favourite deletes the record

---

### 3.5 Submission (Community Translation Suggestion)

**Description:** Tracks community-contributed translation suggestions awaiting review.

| Attribute | Data Type | Nullable | Description |
|-----------|-----------|----------|-------------|
| SubmissionId | int | No | Primary key (identity) |
| UserId | nvarchar(450) | No | Foreign key → AspNetUsers.Id (submitter) |
| PhraseId | int | No | Foreign key → Phrase.PhraseId |
| Language | nvarchar(50) | No | Target language for the suggestion |
| SuggestedText | nvarchar(2000) | No | The suggested translation |
| Status | nvarchar(20) | No | "Pending", "Approved", "Rejected", "CorrectionRequested" |
| RejectionReason | nvarchar(500) | Yes | Reason if rejected |
| CorrectionInstructions | nvarchar(500) | Yes | Instructions if correction requested |
| SubmittedDate | datetime2 | No | When submitted (default: GETDATE()) |
| ReviewedDate | datetime2 | Yes | When reviewed |

**Business Rules:**
- A student can submit multiple suggestions for the same phrase and language
- Pending submissions are not visible in search results
- When approved, a Translation record is created
- When rejected, the student sees the reason on their profile
- When correction requested, the student can resubmit
- Administrators can approve, reject, or request correction

---

### 3.6 LanguageStatistic

**Description:** Tracks usage statistics for analysis and reporting.

| Attribute | Data Type | Nullable | Description |
|-----------|-----------|----------|-------------|
| StatisticId | int | No | Primary key (identity) |
| UserId | nvarchar(450) | Yes | Foreign key → AspNetUsers.Id (null if anonymous) |
| PhraseId | int | Yes | Foreign key → Phrase.PhraseId (null for searches) |
| Language | nvarchar(50) | Yes | Language selected (null if no filter) |
| CategoryId | int | Yes | Foreign key → Category.CategoryId (null if no filter) |
| EventType | nvarchar(20) | No | "View" or "Search" |
| EventDate | datetime2 | No | When event occurred (default: GETDATE()) |

**Business Rules:**
- Every phrase detail view is logged
- Every search is logged (with language and category filters if applied)
- Anonymous users are logged with UserId = null
- Statistics cannot be deleted (append-only log)
- Administrators can view aggregated reports

### 4.1 Relationship Overview

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Category   │─────│    Phrase    │─────│  Translation │
│  (1)         │     │     (M)      │     │     (M)      │
└──────────────┘     └──────────────┘     └──────────────┘
                            │  │                    │
                            │  │                    │
                            │  └──────┐             │
                            │         │             │
                            ▼         ▼             ▼
                    ┌──────────────┐ ┌──────────────┐
                    │  Favourite   │ │  Submission  │
                    │     (M)      │ │     (M)      │
                    └──────────────┘ └──────────────┘
                            │              │
                            │              │
                            ▼              ▼
                    ┌──────────────┐ ┌──────────────┐
                    │     User     │ │     User     │
                    │  (ASP.NET)   │ │  (ASP.NET)   │
                    └──────────────┘ └──────────────┘

┌──────────────┐
│ LanguageStat │
│   istic      │
│              │─── references ─── User (nullable)
│              │─── references ─── Phrase (nullable)
│              │─── references ─── Category (nullable)
└──────────────┘
```

### 4.2 Detailed Relationships

| Relationship | Cardinality | Description |
|--------------|-------------|-------------|
| Category → Phrase | 1 : M | One Category can have many Phrases; one Phrase belongs to exactly one Category |
| Phrase → Translation | 1 : M | One Phrase can have many Translations; one Translation belongs to exactly one Phrase |
| User → Favourite | 1 : M | One User can have many Favourites; one Favourite belongs to exactly one User |
| Phrase → Favourite | 1 : M | One Phrase can be favourited by many Users; one Favourite belongs to exactly one Phrase |
| User → Submission | 1 : M | One User can submit many Translations; one Submission belongs to exactly one User |
| Phrase → Submission | 1 : M | One Phrase can have many Submissions; one Submission belongs to exactly one Phrase |
| User → Translation | 1 : M | One User can submit many Translations (as submitter) |
| User → Translation | 1 : M | One User can review many Translations (as reviewer) |

### 4.3 Foreign Key Constraints

| Foreign Key | Referenced Table | ON DELETE Behavior |
|-------------|------------------|-------------------|
| Phrase.CategoryId | Category.CategoryId | RESTRICT (prevent deletion if phrases exist) |
| Translation.PhraseId | Phrase.PhraseId | CASCADE (delete translations when phrase deleted) |
| Favourite.UserId | AspNetUsers.Id | CASCADE (delete favourites when user deleted) |
| Favourite.PhraseId | Phrase.PhraseId | CASCADE (delete favourites when phrase deleted) |
| Submission.UserId | AspNetUsers.Id | CASCADE (delete submissions when user deleted) |
| Submission.PhraseId | Phrase.PhraseId | CASCADE (delete submissions when phrase deleted) |
| LanguageStatistic.UserId | AspNetUsers.Id | SET NULL (keep stats if user deleted) |
| LanguageStatistic.PhraseId | Phrase.PhraseId | SET NULL (keep stats if phrase deleted) |
| LanguageStatistic.CategoryId | Category.CategoryId | SET NULL (keep stats if category deleted) |

## 6. Indexing Strategy

### 6.1 Performance-Critical Indexes

#### Category Table
```sql
CREATE INDEX IX_Category_Name ON Category(Name);
CREATE INDEX IX_Category_IsActive ON Category(IsActive);
```

#### Phrase Table
```sql
CREATE INDEX IX_Phrase_EnglishText ON Phrase(EnglishText); -- For search
CREATE INDEX IX_Phrase_CategoryId ON Phrase(CategoryId); -- For filtering
CREATE INDEX IX_Phrase_IsActive ON Phrase(IsActive); -- For active filtering
```

#### Translation Table
```sql
CREATE INDEX IX_Translation_PhraseId ON Translation(PhraseId); -- For phrase details
CREATE INDEX IX_Translation_Language ON Translation(Language); -- For language filtering
CREATE INDEX IX_Translation_Status ON Translation(Status); -- For approval filtering
CREATE INDEX IX_Translation_Text ON Translation(Text); -- For search (consider full-text)
```

#### Favourite Table
```sql
CREATE INDEX IX_Favourite_UserId ON Favourite(UserId); -- For user favourites
CREATE UNIQUE INDEX IX_Favourite_Unique ON Favourite(UserId, PhraseId); -- Prevent duplicates
```

#### Submission Table
```sql
CREATE INDEX IX_Submission_UserId ON Submission(UserId); -- For user profile
CREATE INDEX IX_Submission_PhraseId ON Submission(PhraseId); -- For phrase submissions
CREATE INDEX IX_Submission_Status ON Submission(Status); -- For pending list
CREATE INDEX IX_Submission_Language ON Submission(Language); -- For language filtering
```

#### LanguageStatistic Table
```sql
CREATE INDEX IX_LanguageStatistic_User ON LanguageStatistic(UserId);
CREATE INDEX IX_LanguageStatistic_EventDate ON LanguageStatistic(EventDate);
CREATE INDEX IX_LanguageStatistic_Language ON LanguageStatistic(Language);
CREATE INDEX IX_LanguageStatistic_EventType ON LanguageStatistic(EventType);
CREATE INDEX IX_LanguageStatistic_Category ON LanguageStatistic(CategoryId);
```

### 6.2 Full-Text Search Consideration

**Option 1: Standard SQL `LIKE`**
- Simpler to implement
- Works well for small datasets (< 1000 phrases)
- Limited performance for large text searches

**Option 2: Full-Text Indexing**
- Better performance for large datasets
- Supports ranking and relevance
- Requires SQL Server Full-Text Search capability

**Recommendation:** Implement Full-Text Search for production

```sql
-- Full-Text Index Setup
CREATE FULLTEXT CATALOG PhraseCatalog AS DEFAULT;
CREATE FULLTEXT INDEX ON Phrase(EnglishText) 
    KEY INDEX PK_Phrase 
    WITH STOPLIST = SYSTEM;
CREATE FULLTEXT INDEX ON Translation(Text) 
    KEY INDEX PK_Translation 
    WITH STOPLIST = SYSTEM;
```

### 6.3 Indexing Priority Matrix

| Priority | Query Type | Tables | Indexes Needed |
|----------|-----------|--------|----------------|
| High | Search by keyword | Phrase, Translation | EnglishText, Text (full-text) |
| High | Filter by category | Phrase | CategoryId |
| High | Get user favourites | Favourite | UserId |
| High | Get pending submissions | Submission | Status |
| Medium | Get phrase translations | Translation | PhraseId |
| Medium | Get user submissions | Submission | UserId |
| Medium | Get statistics reports | LanguageStatistic | EventDate, Language, CategoryId |
| Low | Category management | Category | Name |

## 7. Data Types & Constraints

### 7.1 Column Data Type Standards

| Column Type | SQL Data Type | .NET Type | Notes |
|-------------|---------------|-----------|-------|
| Primary Key | int | int | Identity column, auto-increment |
| Foreign Key | int | int | Matches referenced PK type |
| User ID | nvarchar(450) | string | ASP.NET Identity standard |
| Names/Titles | nvarchar(100) | string | Limited length, indexed |
| Text Content | nvarchar(1000) | string | Phrase English text |
| Long Text | nvarchar(2000) | string | Translation text, suggestions |
| Description | nvarchar(500) | string | Optional descriptions |
| Language Code | nvarchar(50) | string | Standard language names |
| Status | nvarchar(20) | string | Fixed set of values (consider enum) |
| Reason | nvarchar(500) | string | Rejection/correction reasons |
| Boolean | bit | bool | 0 = false, 1 = true |
| DateTime | datetime2 | DateTime | UTC time, precision to 100ns |
| Nullable Types | appropriate | nullable<T> | For optional fields |

### 7.2 Validation Constraints

#### Check Constraints

```sql
-- Status must be one of: Pending, Approved, Rejected, CorrectionRequested
ALTER TABLE Translation ADD CONSTRAINT CK_Translation_Status 
    CHECK (Status IN ('Pending', 'Approved', 'Rejected'));

-- EventType must be: View or Search
ALTER TABLE LanguageStatistic ADD CONSTRAINT CK_LanguageStatistic_EventType 
    CHECK (EventType IN ('View', 'Search'));
```

#### Unique Constraints

```sql
-- Prevent duplicate category names
ALTER TABLE Category ADD CONSTRAINT UC_Category_Name UNIQUE (Name);

-- Prevent duplicate favourites
ALTER TABLE Favourite ADD CONSTRAINT UC_Favourite_Unique 
    UNIQUE (UserId, PhraseId);

-- Prevent duplicate approved translations
-- (Note: This is enforced in application logic, but database can have a filtered unique index)
CREATE UNIQUE INDEX UX_Translation_UniqueApproved 
    ON Translation(PhraseId, Language) 
    WHERE Status = 'Approved';
```

## 8. Migration & Versioning Strategy

### 8.1 Migration Approach

**Tool:** Entity Framework Core Migrations

**Process:**
1. Create initial migration: `dotnet ef migrations add InitialCreate`

## 10. Open Questions & Decisions

### 10.1 Open Questions

| # | Question | Status | Decision By |
|---|----------|--------|-------------|
| 1 | Should `EnglishText` be unique? | Pending | Team Lead |
| 2 | How should duplicate translation submissions be handled? | Pending | Team Lead |
| 3 | Should we use Soft Delete (IsActive) or Hard Delete for all tables? | ✅ Decided | Soft Delete |
| 4 | Should we implement Full-Text Search or stick with `LIKE`? | Pending | Team Lead |
| 5 | What is the maximum length for `Text` fields? | ✅ Decided | 1000 for Phrase, 2000 for Translation |
| 6 | Should statistics track unauthenticated users? | ✅ Decided | Yes (with UserId = null) |
| 7 | Should categories be fixed or user-configurable? | ✅ Decided | Fixed (Admin-configurable) |
| 8 | What should happen when a phrase is deactivated? | ✅ Decided | Hidden from search, favourites remain |

### 10.2 Decisions Made

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Use Soft Delete (IsActive flag) for all tables | Preserves data integrity and audit trail |
| 2 | Categories are configurable by Admin | Allows flexibility for campus changes |
| 3 | Statistics track both authenticated and anonymous users | Provides complete usage picture |
| 4 | Use `datetime2` for all dates | Better precision, follows standards |
| 5 | Use `nvarchar` for all string columns | Supports Unicode (all 11 languages) |
| 6 | Implement unique constraint on Favourite (UserId + PhraseId) | Prevents duplicate favourites |
| 7 | Filtered unique index on Translation (PhraseId + Language) where Status = 'Approved' | Prevents duplicate approved translations |
| 8 | CASCADE delete for Favourite when User or Phrase is deleted | Maintains referential integrity |
